Start Here¶

Container¶

The quickest way to get started building CCF applications is to use the Build Container.

sudo docker run -it ccfciteam/ccf-app-ci:latest

That contains a release of CCF along with a complete build toolchain, and startup scripts. It can be run on hardware that does not support SGX, in which case you will want to use the virtual binaries, or build in virtual mode.

Note

virtual mode provides no security guarantee. It is only useful for development and prototyping.

If you use Visual Studio Code you can install the Remote Container extension and use the sample devcontainer.json config. More details on that process.

If your hardware does support SGX, and has the appropriate driver installed and loaded, then you will only need to expose the device to the container by passing --device /dev/sgx:/dev/sgx when you start it.

SGX-enabled VM¶

The install directory contains (under getting_started/) the Azure and Ansible scripts required to create and set up an SGX-enabled dev VM in Azure, with all necessary dependencies to build CCF itself or CCF applications.

If you use Visual Studio Code you can install and set up the Remote - SSH extension to connect to your SGX-enabled VM.

Creation¶

These steps require an Azure subscription.

Azure Confidential Compute (ACC) offers DC-series VMs using SGX hardware, which can be deployed either through a Marketplace App, or via the following command line instructions. On Windows, you can use WSL or Azure Cloud Shell (Bash) to run these.

First clone the CCF repository or download a release tarball here. Run the pre_make_vm.sh script to install the Azure CLI and the Open Enclave Engine (oe-engine) that are required to create the DC-series in Azure:

$ cd CCF/getting_started/create_vm
$ ./pre_make_vm.sh # Requires sudo privileges

Then, to get a VM up and running in the East US region, you can run the following command:

$ ./make_vm.sh $AZURE_SUBSCRIPTION_NAME [path_to_ssh_public_key]

After signing in to your Azure account, the script will create a default ccf user on the VM, authenticated by the public key specified by path_to_ssh_public_key (defaults to ~/.ssh/id_rsa.pub).

Setup¶

CCF dependencies include Open Enclave, mbedtls, libuv and libcurl.

To quickly set up the dependencies necessary to build CCF, simply run:

$ cd CCF/getting_started/setup_vm
$ ./run.sh driver.yml ccf-dev.yml

Once this is complete, you can proceed to Building CCF from Source.

On a machine without SGX, you can instead run:

$ cd CCF/getting_started/setup_vm
$ ./run.sh ccf-dev.yml

Next Steps¶

Once your setup is complete, you may want to get familiar with some of CCF’s Concepts. You will then be able to:

  1. Create a consortium and agree on the constitution

  2. Develop a CCF application, based on the example logging application

  3. Start a new CCF network to deploy the application

  4. Let the consortium configure and open the network to users

  5. Have users issue business transactions to the application

Requirements¶

At the moment, CCF only builds and runs on Linux. It is primarily developed and tested on Ubuntu 18.04. Running CCF with full security guarantees requires SGX hardware with FLC.

Releases¶

CCF releases (ccf.tar.gz) are available on the GitHub repository release page. Once downloaded, the extracted install directory can be copied to a long-lived path, e.g. under /opt/.

To setup a CI for your CCF application, you may want to use the Build Container. If you are running CCF nodes in containers, the Runtime Container is a good place to start.
